Using Git & GitHub in Linux Terminals and Desktops.

Alex Heilman
January 4, 2023
Computer

Git is a popular version control system; in which any contributor mirrors the entire relevant code-base on their own respective device. GitHub is their assocated internet hosting service, where remote repositories may be shared.

In software development, it is generally advantageous to have some means by which to track changes and previous versions of the software through development. By far, the most popular version control software is git, which is available as both a desktop client (on Windows and MacOS) and an in-terminal program (on all platforms). Along with this version control software is git's native internet hosting service GitHub, which is a convenient place to store repositories remotely for common access by different developers and devices.

Below, a brief overview of git's structure and usage, specfically in concert with remote repositories on GitHub, is given for the most common use cases. For a better overview/reference of usage see Roger Dudler's Guide.

Git Operations: Overview

The current version of some program is deemed the main branch, where the current main branch is effectively the current version of whatever software is being developed or built on.

Git Operations: Usage

The most basic usage of git occurs when there is only the main branch and no new files are being added, only changed. In this case, one first need to commit their changes, and then push them to the main branch.

Updating the Local Repository

To first initialize a current directory as a git repository, give the following command in the desired directory: git init If you instead wish to clone a current git repository on your local device, use the following: git clone <uri> where <uri> should be replaced with the unique identifier pointing to the desired git repository (such as the https address of github repository). Note you may also clone a local repository via the same command by replacing the uri with a local directory that contains a git repository.

If you wish to pull new changes on the remote repository associated with the local one, give the following command in the local directory containing the repository: git pull If you have local changes that conflict with these remote changes, the simple form above may return an error.

Updating the Main Repository

The very first thing that needs to be done before updating anything is to add the local changes to the index. git add * where the asterisk denotes to add all changes to the index. This asterisk may be replaced with a list of file names to specify which changed files should be added to the index.

To commit these changes to the local HEAD, you must then use the command below (note that a messsage associated with the changes detailing those made in the new commit is mandatory). git commit -m "<commit message>"

These changes may then be pushed to the remote data-base and established as the new version of the current branch (by default main) via the command below. git push origin master

Additional Fields

Some optional fields are also commonly used with the SSH command in terminal, these include specfiying the port number with -p to connect via (which is 22 by default), whether or not to show extra data about the connection (which is supressed by default) with -v. The usual -h can also be used to display additional options, as usual.

Server Side

As mentioned above, the remote server one is attempting to connect to and control requires that it is acting as an SSH server. The method of initiating this service is varied depending on the SSH client chosen, but the a common case (OpenSSH in Arch Linux terminal) is covered here.

Server Initialization

Regardless of the specifics, setting up an SSH server generally requires the intended server to treat an SSH event listener, such as SSHD, as a service that runs constantly in the background, listening for requests through the network for SSH connections. Starting this service thus constitutes the intitialization of a device as an SSH server and is all one needs to do to set it up as such.

SCP Protocol: Overview

Another common operation one often encounters is that of secure transfer of files from one device to another over an untrusted network. This may be accomplished with the use of a transfer protocol based on SSH such as the Secure File Transfer Protocol (SFTP) or Secure Copy Protocol (SCP), which is discussed here.

SCP Protocol: Usage

The most basic usage of SCP is to copy a file from a local device to a remote server, or vice versa. This may be specficied in the addresses of the files. SCP takes as the first argument the address of the file to be copied and then takes the last argument as the address for the file to be copied to. scp <current file address> <copy destination> Note that one may include several files to be copied, which will be read as all but the last argument given to scp (where the last is always the destination). Hence, to copy multiple files simultaneously to the same directory, the following structure of command should be given. scp <current file address 1> ... <current file address n> <copy destination> The above will then copy files from <file address 1> to <file address n> to the directory specified at the address <copy destination>.

A common problem encountered when using SCP is when one wishes to copy whole directories from one location to another. This case may be handled by applying the additional, optional argument -r (for recursive) letting SCP know you wish to copy those directories recursively to the destination.
